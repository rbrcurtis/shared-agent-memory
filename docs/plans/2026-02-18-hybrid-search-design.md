# Hybrid Search (Dense + BM25) for Shared Memory MCP

## Problem

Search relies only on vector similarity (semantic search). Exact keyword matches — ticket IDs like "TRK-4378", function names like "useRequest", config values — are unreliable because embeddings don't capture specific tokens well. A search for "TRK-4378" might not surface the relevant memory if the vector similarity is low.

## Solution

Add BM25 keyword search alongside dense vector search. Combine results using Qdrant's server-side Reciprocal Rank Fusion (RRF). Memories that match both semantically and by keyword rank highest.

## Approach: Qdrant-native Prefetch + RRF Fusion

Send a single Qdrant query with two prefetch branches (dense + BM25). Qdrant fuses results server-side via RRF, eliminating the need for client-side score normalization. BM25 tokenization also happens server-side via Qdrant's built-in `Qdrant/bm25` inference model.

## Collection Schema Change

Current:
```
vectors: { size: 384, distance: Cosine }
```

New:
```
vectors:
  dense: { size: 384, distance: Cosine }
sparse_vectors:
  text-bm25: { modifier: idf }
```

Cannot add sparse vectors to existing collection — requires migration (recreate + re-insert).

## Search Flow

1. Agent calls `search_memory("TRK-4378 scheduling", limit=10)`
2. Daemon embeds query → 384-dim dense vector
3. Daemon sends single Qdrant `query` with two `prefetch` branches:
   - Dense: query with dense vector, `using: "dense"`, limit = requested * 3
   - BM25: query with `{text: query, model: "Qdrant/bm25"}`, `using: "text-bm25"`, limit = requested * 3
   - Both branches include tombstone `is_empty` filter
4. Top-level: `query: {fusion: "rrf"}` — Qdrant combines via Reciprocal Rank Fusion
5. Daemon applies forgetting curve: `final_score = rrf_score * retention`
6. Tombstone decayed results, reinforce returned results (unchanged from forgetting curve)

## Store/Update Flow

Points upserted with both vectors:
- `dense`: generated by local Xenova embedding model (unchanged)
- `text-bm25`: `{text: memory_text, model: "Qdrant/bm25"}` — server-side BM25 sparse vector

## Migration

1. Snapshot existing collection as backup
2. Scroll all points from old collection (with vectors + payloads)
3. Create new collection with dense + sparse config
4. Re-insert all points: original dense vector → named `dense`, text → BM25 via server-side model
5. Verify point count matches
6. Runs against both localhost:6333 and ${QDRANT_HOST}:443
